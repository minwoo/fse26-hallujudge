Motivation

LLMs review code at scale — but can their feedback be trusted?

  • LLM reviewers are now in production: Atlassian's RovoDev Code Reviewer serves 4,000+ engineers — 40,000+ comments / month across 2,500 repositories.
  • They produce fluent, plausible comments — and some are hallucinated: ungrounded, irrelevant, or misaligned with the actual code change.
  • Hallucinated feedback erodes trust, causes confusion, and hinders adoption.
  • Unlike code generation, review hallucinations can't be caught by a compiler or runtime — they must be judged.
The risk isn't wrong code.
It's confident, wrong feedback reaching a developer who must decide whether to believe it.
The Problem

Context misalignment

A type of intrinsic error (contradiction with the input), where comments maybe syntactically valid and stylistically appropriate, but semantically inconsistent with the actual code change.

THE CODE DIFF UNDER REVIEW

12 public Response fetch(Request r) {
13- return client.call(r);
13+ return client.callAsync(r).join();
14 }

THE GENERATED CODE REVIEW COMMENT

“This change may expose a SQL injection vulnerability — validate and sanitize the query input.”
Ungrounded. No SQL query was modified anywhere in the diff. The claim is fluent, but nothing in the change entails it.
A claim is grounded only if the diff fully entails it — partial correctness is insufficient. A review is hallucinated when it contains ≥ 1 ungrounded claim.
The Gap

Why existing evaluation falls short

Reference-based
e.g., BLEU, Exact Match

Measure lexical overlap, not semantic grounding. Requires a gold reference comment that rarely exists. Cannot evaluate generated code reviews that deviate from the gold reference.

Traditional Reference-free
e.g., feature attribution, uncertainty metrics

Have been shown in past studies to perform poorly at detecting hallucinations in generated code review comments.

Manual review
Human annotation

Expensive, subjective, and difficult to reproduce at scale.

We need a highly parameterized reference-free judge that is capable of reasoning about whether each claim is grounded in the code change itself.
Approach

HalluJudge — hallucination as a context-alignment problem

Inputs
Code Diff + Generated Review Comment
→
HalluJudge
Assessment Strategy × LLM Judges
→
Output
Alignment Score + Evidence-based Explanation
  • Reference-free — no gold comment required.
  • Asks: is each claim in the code review comment supported by, traceable to, and consistent with factual elements in the code change?
  • Grounding: a claim is grounded only if the code diff fully entails it (partial ≠ grounded).
  • Hallucination: a review with ≥ 1 ungrounded claim.
Scoring

A 0–4 alignment scale

An operational rubric to help the Judge reason through the decision.

0
Fully aligned
All claims directly supported & consistent
1
Mostly aligned
Most claims are supported, minor inconsistencies
2
Partially aligned
Some claims are supported, others are not
3
Mostly misaligned
Very little support from the code diff
4
Completely misaligned
Claims are unsupported or contradicted by the code diff
Score 0 → grounded, non-hallucinated. We enforce a strict zero-tolerance policy toward inconsistencies in the generated review.
Scores 1–4 → increasing misalignment; any ungrounded claim marks the review hallucinated.
Approach

Four assessment strategies

All reference-free — they differ in reasoning depth versus cost.

1 · Directzero-shot

Out-of-the-box reasoning with system + user prompt only. Lowest cost.

2 · Few-Shot5 examples

Five labeled examples - one per score level (0–4) to sharpen decision boundaries.

3 · Multi-Step Reasoningchain-of-thought

Guided reasoning - comprehension, traceability, support, conflict & relevance, completeness, final judgment.

4 · Tree of Thoughtsparallel ★

Four complementary hypothesis branches, then converge on the strongest evidence path.

Approach · Best performer

Tree of Thoughts

This strategy explicitly instructs an LLM to explore multiple reasoning paths in parallel and evaluate hypotheses before reaching the conclusion. Instead of following a single linear reasoning chain, the model branches into several possible lines of thought, reflects on their validity, and selects the most coherent path.

Research Questions

Three questions covering different aspects

RQ1
Effectiveness  
To what degree do the assessment strategies in HalluJudge effectively detect hallucinations in code review comments? (Precision · Recall · F1)
RQ2
Efficiency  
How efficient is HalluJudge in detecting code review hallucination? (Tokens · $USD)
RQ3
Alignment  
To what degree do HalluJudge’s hallucination judgments align with developers’ preferences in practice? (Consistency · Coverage)
RQ1 · Effectiveness
Best configuration: Tree of Thoughts w/ Gemini 3 Pro achieves an F1 of 0.85.
  • Across Both Models - Tree of Thoughts ranks 1st, Direct ranks 2nd.
  • Gemini 3 Pro is more stable across strategies (F1 ranges between 0.06 vs 0.09); variation is driven mainly by recall.
RQ2 · Efficiency
Trade-off: Direct is the cheapest; Tree of Thoughts is the most accurate at the highest cost.
  • On average, direct uses 2.7k–3.3k input and 81–89 output tokens, costing $0.004-$0.009 per inference.
  • Tree of Thoughts buys accuracy with extra reasoning tokens, costing an additional $0.005-$0.007 on average per inference.

Implications

Why it matters

A practical safeguard

Drop HalluJudge in as a gating layer to keep plausible-but-wrong comments away from developers — addressing a key barrier to adoption and trust.

Cost-effective & explainable

As low as $0.009 per judgement — with interpretable, evidence-based reasons behind every judgment.
