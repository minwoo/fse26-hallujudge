Motivation

LLMs review code at scale — but can their feedback be trusted?

  • LLM reviewers are now in production: Atlassian's RovoDev Code Reviewer serves 4,000+ engineers — 40,000+ comments / month across 2,500 repositories.
  • They produce fluent, plausible comments — and some are hallucinated: ungrounded, irrelevant, or misaligned with the actual code change.
  • Hallucinated feedback erodes trust, causes confusion, and hinders adoption.
  • Unlike code generation, review hallucinations can't be caught by a compiler or runtime — they must be judged.
The risk isn't wrong code.
It's confident, wrong feedback reaching a developer who must decide whether to believe it.
The Problem

Context misalignment

Comments that are syntactically valid and stylistically appropriate — yet semantically inconsistent with the code change.

THE DIFF — sync → async conversion

12 public Response fetch(Request r) {
13- return client.call(r);
13+ return client.callAsync(r).join();
14 }

THE GENERATED COMMENT

“This change may expose a SQL injection vulnerability — validate and sanitize the query input.”
Ungrounded. No SQL query was modified anywhere in the diff. The claim is fluent, but nothing in the change entails it.
A claim is grounded only if the diff fully entails it — partial correctness is insufficient. A review is hallucinated when it contains ≥ 1 ungrounded claim.
The Gap

Why existing evaluation falls short

Reference-based
BLEU · Exact Match

Measure lexical overlap, not semantic grounding — and require a gold reference comment that rarely exists.

Reference-free NLP
Generic metrics

Perform poorly at detecting hallucinations in code-review comments.

Manual review
Human annotation

Expensive, subjective, and difficult to reproduce at scale.

We need a reference-free judge that reasons about whether each claim is grounded in the code change itself.
Approach

HalluJudge — hallucination as a context-alignment problem

Inputs
Code diff + generated review comment
→
HalluJudge
Assessment strategy × LLM judge
→
Output
Alignment score 0–4 + evidence-based explanation
  • Reference-free — no gold comment required.
  • Asks: is each claim supported by, traceable to, and consistent with the code change?
  • Grounding: a claim is grounded only if the diff fully entails it (partial ≠ grounded).
  • Hallucination: a review with ≥ 1 ungrounded claim.
Scoring

A 0–4 alignment scale

0
Fully aligned
All claims directly supported & consistent
1
Mostly aligned
Minor inconsistencies present
2
Partially aligned
Mixed support
3
Mostly misaligned
Minimal support
4
Completely misaligned
Unsupported or contradicted
Score 0 → grounded, non-hallucinated.
Scores 1–4 → increasing misalignment; any ungrounded claim marks the review hallucinated.
Approach

Four assessment strategies

All reference-free — they differ in reasoning depth versus cost.

1 · Directzero-shot

Out-of-the-box reasoning with system + user prompt only. Lowest cost.

2 · Few-Shot5 examples

Five labeled examples — one per score level (0–4) — to sharpen decision boundaries.

3 · Multi-Step Reasoningchain-of-thought

Comprehension → traceability → support → conflict/relevance → completeness → judgment.

4 · Tree of Thoughtsparallel ★

Four complementary hypothesis branches, then converge on the strongest evidence path.

Approach · Best performer

Tree of Thoughts — reason in parallel, then converge

Click a branch to expand. Converge via evidence weight · contradiction analysis · confidence · strongest-evidence path.

1Alignment Hypothesis+
Identify the diff elements that support each claim in the comment.
2Misalignment Hypothesis+
Find unsupported claims, contradictions, or assumptions drawn from outside the change.
3Evidence Mapping+
Build explicit trace links between each comment sentence and specific diff elements.
4Context Boundaries+
Verify every claim stays within the intended scope of the change.

Discussion

What we learned

1 · Ensembling doesn't help

Averaging all four strategies scores F1 0.81 (Gemini 3) and 0.73 (GPT-5.1) — both below Tree of Thoughts alone (0.85). The strategies don't provide complementary signals.

2 · Judgments are evidence-grounded

HalluJudge flags an invalid isPresent() suggestion by citing length() in the diff — the return type is String, not Optional. Tree-of-Thoughts branches add causal analysis the direct strategy misses.

Implications

Why it matters

A practical safeguard

Drop HalluJudge in as a filtering / validation layer to keep plausible-but-wrong comments away from developers — directly addressing a key barrier to adoption and trust.

Cost-effective & explainable

A $0.009 alternative to expensive human annotation — with interpretable, evidence-based reasons behind every judgment.
